THCC: Compiling Linear Regression onto a Homemade CPU
THCC is a compiler I built in Haskell that takes C-style source language and compiles it down to machine code for THMM which is a custom 16-bit accumulator CPU I created for a computer organization class during my undergraduate at Colorado College. The idea is that every layer of the stack is simple enough that you can read through the entire thing and understand what's going on. The demo payload computes linear regression using the closed-form normal equation and produces "w = 2", "b = 1" as expected.
Goals Achieved
THMM is a CPU I originally built in Logisim-Evolution from scratch using logic gates. It has 13 opcodes, 256 words of shared code/data RAM, and a single accumulator register. For this project, I made a few modifications to it. I added signed two's complement arithmetic because the regression computation needs negative intermediate values. I also added the "addn" instruction partway because I noticed that the compiler was wasting a lot of instructions materializing literals to temporary RAM slots just to add them. On top of that, I wrote a Python simulator ("cpu.py") where every variable in the "tick()" function maps one-to-one to a labeled wire on the datapath schematic. All the signals are represented as bit strings instead of integers, which makes the simulator directly simulate what is actually happening on the wires. 
What I did for this project is build THCC, the compiler. I built it in Haskell using Megaparsec for parsing. The grammar matches what I specified in my proposal. It supports "int" variable declarations with initializers, arithmetic with "+ - * /" and standard C precedence, parentheses, and line comments. The compiler pipeline also matches what I specified in my proposal: parsing with correct precedence using "makeExprParser", then code generation that first collects all variable declarations, then emits symbolic instructions, and finally a linker that uses the total instruction count to assign concrete RAM addresses to variables and temporaries. One of the main challenges was handling the fact that THMM only has one register, so in the general case intermediate values in an expression have to get spilled to a temporary RAM address. The codegen handles this by recursively generating code for subexpressions and storing results to temporaries before combining them. I also ended up implementing two optimizations that weren't in the original plan: when the right operand of a binary operation is a variable, the compiler skips the temporary and uses the memory-form instruction directly (like "addm" or "mulm" against that variable's address), and when you're adding a literal it uses the "addn" immediate instruction instead of storing the literal to a temp first. 
I implemented simple linear regression demo to show that it works end-to-end. The source file declares three data points that fall exactly on the line "y = 2x + 1", computes the sums needed for the normal equation, then plugs those into the formula to solve for the slope "w" and intercept "b". The compiler turns this into 83 machine instructions. After loading those into the simulator and running it, the values "w = 2" and "b = 1" show up in RAM at the addresses the compiler reports. I committed the expected assembly and hex outputs as fixture files so any regressions would be immediately visible in version control. 
For tests the Haskell compiler has 29 hspec test examples across five spec modules covering lexer helpers, parser behavior, code generation, full pipeline integration, and QuickCheck property-based tests. The Python simulator has 45 tests organized in layers from combinational truth tables up through full program end-to-end tests, including one that compiles THCC source with the Haskell compiler and verifies the regression output on the simulator. Haddock documentation is at 100% across all modules.
Challenges Remain
My proposal specified three separate Haskell modules as compiler stages: a Lexer, a Parser, and a Code Generator. The idea was that the Lexer would produce an explicit token list using a "Token" data type with constructors like "KW_Int", "Ident", "IntLit", and so on, and then the Parser would consume that list to build the AST. Once I started actually working with Megaparsec though, I realized this separation was unnecessary. Megaparsec has built-in lexer combinators like "L.lexeme" and "L.symbol" that handle whitespace and tokenization as part of parsing, so I never needed to produce an intermediate list of tokens at all. Having a separate "Token" type and a standalone Lexer module would have just been boilerplate that made the code harder to follow without adding any real value. So I collapsed the lexer into the parser and let Megaparsec handle everything in one pass. The lexer-level helpers still exist as exported functions and are tested in their own spec file, so the functionality is all there, it's just not a separate compilation stage. 
I also changed the directory layout from what I originally planned. I was going to use a nested "src/THCC/" directory tree, but with only four library modules it felt like overkill. Flattening everything to top-level ".hs" files makes it easier to navigate and stays true to the "simple enough to hold in your head" philosophy.
Beyond the plan changes, there are known limitations that were part of the project scope from the start. The language has no control flow, no functions, no arrays, no unary minus, and no I/O. Literals are limited to 0 through 255. There's only one type ("int", 16-bit signed). On the hardware side the only conditional is "goif0" which branches when the accumulator equals zero. There's no shift or rotate and no indirect memory addressing. These were all intentional. The project was scoped to straight-line arithmetic on purpose so the entire stack stays small enough to understand.
Future Works and Roadmap
As I mentioned in my proposal, the simplicity of the project is both a limitation and an opportunity. Students could implement these extensions themselves as a learning exercise. The most natural next step would be adding "if" and "while" to the language since the ISA already has "goif0" which could support simple branching. After that, functions with a software stack convention and arrays once indirect addressing gets added to the hardware. On the compiler side, constant folding, dead store elimination, and reusing temporary slots across statements are all low-hanging optimization passes. A web-based trace viewer that consumes the simulator's bit-string state would make the educational value a lot more accessible, and rebuilding the datapath in Logisim-Evolution validated cycle-for-cycle against the Python simulator would close the loop back to real hardware. The big stretch goal would be implementing gradient descent on the same regression problem so students could compare the closed-form solution with an iterative one and see the tradeoffs.
